The bottleneck was never capacity. It was always context.

Story points, sprint planning, and velocity metrics were designed for a world where shipping software took weeks per feature. That world is ending. AI-native engineering changes delivery time from weeks to hours. But most teams are still planning in points and running weekly ceremonies built for a cadence that no longer exists.

The overhead has not scaled down with the delivery time. It has stayed the same, or grown. That is the problem context-first is trying to solve. Sessions-not-stories is the engineering-specific instantiation of the principle — the unit of attention you actually plan, build, and ship inside.

AI-native engineering amplifies engineering judgment. Sessions give that judgment a repeatable structure. Together they produce a compounding effect that neither delivers alone.

The session primitive

A session is what one engineer and an AI coding assistant can design, build, test, and ship before context drift sets in. Typically 60–120 minutes. Always ends with a coherent artifact.

Old primitive: story New primitive: session
Sized in pointsSized in focus minutes
Bounded by sprint calendarBounded by cognitive context
Accepted via demoAccepted via tests + tag + repo memory
Not replayableReplayable — chat, git, memory
Backlog growsCompounds — each session paves the next

Bounded by cognition, not calendar

A 90-minute session with a sharp goal beats two days of fragmented work. The constraint is attention, not capacity.

Always produces an artifact

Green tests, FF-merge, tag, repo memory updated. Sessions that don't close cleanly are a smell.

Replayable by design

The chat transcript, repo memory, and git history are the audit trail. Stories were never replayable; sessions are.

Compounds over time

What repo memory captured this session makes the next session faster. Stories don't compound; backlogs just get longer.

Sessions, Not Stories

Planning shifts from point-sized stories to focus-sized sessions: bounded by cognitive context, accepted through tests, commits, tags, and repo memory, and designed to compound across runs. The PM and engineering job becomes framing the next coherent session, not negotiating abstract backlog units.

Ceremonies as Sessions

Agile ceremonies compress when session hygiene makes intent, progress, and decisions visible before the meeting starts. Code review, retros, planning, demos, and office hours become scoped sessions with clear exit artifacts instead of recurring status rituals.

The first experiment

The session model was not designed upfront. It emerged from a real project — accidentally — and produced results that were hard to ignore. This repo exists to make it intentional, and to replicate it across more engineers, domains, and experience levels.

cLLM — Experiment 001

A synthetic GPU / vLLM simulator built in 4 days by one Partner FDE using GitHub Copilot and an AI-native engineering workflow. 125 commits in 12 days. ~19.5K lines of Go. ~40% test ratio. Race-clean. 3-layer observability shipped day one. GitOps-deployed on K3s / Flux. Comparable engineering bar to a 26-week, 4-person project from 2020.

The session model emerged from that project without being planned. The next experiments will be intentional — logged, measured, and contributed back here.

Run your own experiment

The methodology gets better when more engineers stress-test it across different domains, experience levels, and team contexts. One well-documented experiment is worth more than five opinions.

To contribute: run at least 5 intentional sessions using the session log template, write a summary of what you built and what you learned, and open a PR to the experiments/ folder.

A session log that shows where the model broke down is as valuable as one that shows it working — maybe more.
